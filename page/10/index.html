LeetCode.70.爬楼梯（简单）

发表于 2019-10-31 更新于 2021-09-14 分类于 LeetCode Disqus：

题目

LeetCode.70.爬楼梯（简单）

假设你正在爬楼梯。需要 n 阶你才能到达楼顶。
每次你可以爬 1 或 2 个台阶。你有多少种不同的方法可以爬到楼顶呢？
注意：给定 n 是一个正整数。

题解

思路

最优化问题，考虑用动态规划，看是否符合动态规划的条件。
一般从最终状态做倒推，拆解当前状态问题为上一个状态+有限的步骤。

倒推

爬到第n阶，最后一步一定是只有两种可能：

  1. 先爬到第n-1个台阶，再爬1个台阶
  2. 先爬到第n-2个台阶，再爬2个台阶

把两种可能的方法数累加就是爬到第n阶第方法数。

边界处理

爬到第1阶，只有1种爬法，即爬1个台阶。
爬到第2阶，可以从第0阶爬2个台阶，也可以从第1阶爬1个台阶，共2种爬法。

Java8 Lambda表达式

发表于 2019-10-25 更新于 2021-09-14 分类于 Java Disqus：

Java8 Lambda表达式有什么限制？

Java8的lambda和匿名类可以实现类似闭包的功能：它们可以作为参数传递给方法，并且可以访问其作用域之外的变量。

但有一个限制：它们不能修改定义Lambda方法的局部变量的内容。这些变量必须是隐式final的。

之所以会有这些限制，主要是由于局部变量是保存在栈上的，如果Lambda可以直接访问局部变量，且Lambda是在另一个线程使用的，那么使用Lambda的线程可能会在分配该变量的线程将这个变量收回之后，去访问该变量。因此，Java在访问自由局部变量时，实际上访问的是它的副本，而如果局部变量仅会赋值一次那就没什么区别了。

如果允许捕获可变的局部变量，就会引发造成线程不安全的可能性，而这是我们极力希望避免的。基于种种考虑，Java直接在语法层面做出限制，这其实也是不鼓励程序员使用改变外部变量的典型的命令式编程模式。

Java8 Lambda表达式是怎么实现的？

对lambda表达式里的代码，生成静态私有方法，创建一个函数式接口类的实例来调用这个私有静态方法。

这种方式实现的好处在于：

  1. 使用预定义的函数式接口类，不用新增类，避免新类的加载、验证、解析、初始化的耗时。
  2. lambda表达式代码翻译到字节码的过程，可以动态的改变，方便做统一的优化。

为什么Java 8的Lambda表达式的字节码指令要使用invokedynamic指令？

Java8的设计者之一Brian Goetz对于lambda表达式的设计文档中所提到的方案选择：

  • 一是转换策略要足够灵活以能够应对将来的优化。
  • 二是要维持现有class文件标准的稳定性。

用invokedynamic指令干了什么？

invokedynamic调用JDK里的LambdaMetafactory类的方法来把Lambda表达式翻译为字节码。

不用invokedynamic的坏处？

如果把Lambda表达式的翻译方式写死在类的字节码，以后不方便统一修改翻译策略了。

Java8 Lambda表达式为什么不用匿名内部类实现？

Lambda表达式在语法上看起来就跟匿名内部类一样。

但是对于匿名内部类：

  1. 编译器会为每个匿名类生成一个新的class文件。由于每个class文件在使用前都需要经过一系列的加载、验证、解析、初始化的过程，大量的类文件会直接影响应用的启动性能。
  2. 每个新的匿名类都会为类或者接口产生一个新的子类型。如果你为了实现一个比较器，使用了一百多个不同的lambda表达式，这意味着该比较器会有一百多种不同的子类型。这种情况下，JVM的运行时性能调优会变得更加困难。

为什么Java8的特性需要Android Gradle插件特定版本才能支持？

Android系统上，Java-Bytecode（JVM字节码）是不能直接运行在Android系统上的，需要转换成Android-Bytecode（Dalvik/ART 字节码）。

为什么呢？比如Java 8的Lambda表达式是通过invokedynamic指令实现的，Android的dex编译器不支持invokedynamic指令，导致Android不能直接支持Java 8。

既然不能直接支持，那就只能在Java-Bytecode转换到Android-Bytecode这一过程中想办法，间接支持。这个间接支持的过程我们统称为Desugar（脱糖）过程。

具体是怎么实现的呢？比如Java 8的Lambda表达式，javac编译过后，Lambda表达式里的逻辑会被编译为私有静态方法，然后用invokedynamic去在运行时动态创建一个类实例，每个Lambda表达式都会对应一个函数式类接口的，在类方法里调用静态方法。脱糖的过程，就是把invokedynamic指令去除，直接调用静态方法。

脱糖

脱糖即在编译阶段将在语法层面一些底层字节码不支持的特性转换为基础的字节码结构，(比如 List 上的泛型脱糖后在字节码层面实际为 Object)； Android 工具链对 Java8 语法特性脱糖的过程可谓丰富多彩，当然他们的最终目的是一致的：使新的语法可以在所有的设备上运行。

D8编译器 已经支持脱糖，让 Java 8 提供的特性（如 lambdas）可以转换成 Java 7 特性。

D8、R8编译器有什么特别？

D8 编译器作为默认的 DEX 字节码文件编译器，具有更好的性能；

R8 作为 ProGuard 的替代工具，用于代码的压缩（shrinking）和混淆（obfuscation）；

Java 7-8-9 等等新引入的语言特性并不能直接就能用在 Android 开发中，基本上现在的所有的 Android 开发者还在被困在 Java 6 SE 上。

为了让我们能使用上 Java 8 的特性，Google 使用了 Transformation 来增加了一步编译过程叫 desugaring，其实就是将我们代码里使用的 java 8 新特性翻译为 Dalvik/ART 能够识别的 java 6 字节码。这不可避免会导致一个问题 - 更长的编译时间。

为了解决这个问题，在 Android Studio 3.2 中，Google 使用 D8 替换了旧的 dx 编译器。D8 的主要改进是消除 desuguaring 的过程，让其成为 dex 编译的一部分，从而加快编译速度。

测试驱动开发与单元测试的价值

发表于 2019-09-15 更新于 2021-09-14 分类于 软件测试 Disqus：

衡量一个东西的价值，要看它解决了什么问题。

测试驱动开发的价值

结论：

对于项目结果而言，TDD带来的最大价值是消除了不确定性，保证项目能准确的按照预期进行下去。

对于开发而言，会有更少的调试时间、更少的回归错误、可以更放心的修改代码，编码效率指数级提升，因为只需要理解最少的信息就可以迅速完成任务。

对于测试而言，从低效率手工重复测试工作中解放出来，把精力放在更专业化的兼容性测试、压力测试、性能测试等工作上。

对于产品而言，可以很快看到结果，快速验证想法，迅速得到反馈做出调整。

测试过程的沉淀

对于个人而言，单元测试相当于通过编写代码对功能逻辑进行自测，可以做到：

  • 有及时的测试反馈，通过mock数据环境，快速验证逻辑正确性。
  • 明确所有已知的边界情况，提前规避错误的发生。
  • 沉淀测试过程，测试代码即文档。
  • 测试代码可以自动化执行，别人任何人想验证功能正确性是不需要再花时间了解功能细节的，节约时间。

如果接下来如果尝试给现有的业务代码编写单元测试，会发现测试代码会有大量的冗余和重复，不得不复制粘贴很多重复的测试代码，非常痛苦。

为什么会产生大量重复的测试代码？

因为原来的业务代码，本身就是重复冗余的，没有抽象复用，设计不合理，甚至往往产品需求都没有梳理完整，边界情况都没有考虑全，自然配套的自然测试代码也跟着不合理。

怎样可以不产生重复的测试代码？

那就应该把原来的业务代码，重写为可复用的、标准化、模块化的组织形式。

写到最后，会发现业务代码就完全符合开闭原则了，没有刻意去用什么设计模式，但是代码自然就遵循了所有的设计模式。

那么与其每次重写代码，为什么不在一开始就设计好代码结构？

源头上的诉求

为什么一开始的业务代码没有做到标准化、模块化？

代码是产品设计的实现，如果是因为产品需求、UI设计就没有标准化、模块化、可复用，那么代码就无法标准化。

每个功能的开发人员必然会按照各自理解封装自己的一套逻辑，都是重复劳动。

再加上Android、iOS这种多端开发都是同一套UI，重复劳动的损耗又要多一倍。

要想效率最高，产品设计一开始就一定要标准化、模块化，明确业务的每个细节，用便于下游理解和处理的形式文档化，比如用测试用例的形式描述需求。 这样在理想情况下，可以通过工具把产品设计侧提供的标准化的测试用例直接转换为开发侧的测试代码模板，减小了开发编写测试用例的负担。

开发如果没有去填充测试代码，CI跑测试就会报错，不能允许出包，这就对开发做了强制约束，降低了需求遗漏和理解不一致的可能性。

在Android、iOS多端开发下，信息都能第一时间及时同步，避免了多端实现不一致。

看似增加了产品和设计的工作量，其实最终是节省了所有人的时间，上下游所有人都将受益。否则巨大的沟通成本和信息不同步会把所有人互相消耗的精疲力尽，结果问题还没有从根本上解决。

把源头的问题解决好后，接下来可以观察：

业务代码将会如何演变？

可以发现，为了不让测试代码重复，最终会倒逼业务代码必须要完全符合开闭原则。

并没有刻意去用什么设计模式，代码自然的就遵循了所有的设计模式。

分层架构

代码必须要分层架构，否则代码就不能复用，一点小改动就会影响到很多其他模块。

一个常见的例子是，很多人喜欢把UI要用的实体类、数据库用的实体类、网络用的实体类三者合一使用同一个类，因为三种场景下属性含义有重复，定义三个类看起来麻烦了，但对于模块化是不方便的：

  • 一旦你要把网络、数据库、UI的代码独立抽离出来，这一个实体类还要再复制两份。
  • 对于单独的模块，这个总的实体类中大概率都会有用不到的属性代码，不删除就只会增加理解成本、增加程序体积。
  • 实体类的往往还有一堆注解，是网络请求所使用的序列化框架的注解和数据库ORM框架的注解，所以你复制完后，还要去掉注解，否则就得让数据库模块去依赖序列化框架，或者网络模块去依赖ORM框架。

这只是一个实体类带来的重构问题，如果有几百个类都是这样，重构阻碍就放大了很多倍，所以在一开始就要按分层模块化的标准来编写代码。

带UI的页面要用MVVM架构，因为VM和V自动映射，映射的过程是框架写好的、是标准化的，不需要自己手写；如果是MVP结构，就要手写UI和数据的映射代码，只要手写了，你就得写测试代码覆盖这些逻辑，所以MVVM是比MVP少一些模板化的测试代码的。

单一职责与依赖注入

类之间的职责必须单一。

比如类A如果要使用到类B和类C中的方法，你是不能在类A里直接创建类B和类C的，否则你无法对类A的方法进行模拟测试，因为无法创建B和C的模拟类传给A使用。

解决这个问题就必须要做依赖注入，把类B和类C的创建移出类A之外，通过set、构造函数等方式传递进来，保证在测试时可以随时替换类B和类C。

可以发现只是为了方便测试做了类隔离，也方便了在开发中替换B和C的实现，代码重构代价更小了。

当依赖注入的需求变得复杂，比如要做生命周期管理、状态共享，就需要借助依赖注入框架来完成。

壳化平台特定的代码

在Android中，为了能在JVM上快速运行测试，还会把所有Android相关的类访问都做一层代理封装，这样强迫了代码隔离。并且封装的API是语义化的，业务代码又增加了可读性。

Android的类主要充当容器和交互的作用，所有的业务逻辑都可以在JVM上直接模拟测试，是非常方便的。

Android代码壳化后，理论上非Android相关的代码应该是可以跨端使用的，只要有工具能做到。

我们可以简单的把没有重复测试代码的业务代码，称为高可测性的代码。

如何加速测试代码的编写？

反过来思考，阻碍测试代码编写的原因还有哪些？把这些影响因素都解决了就好了。

提前做标准化设计

之前说了如果业务代码架构设计不合理，复用性很差，测试代码会很难写，要被迫修改业务代码，重构是非常大的成本。

如果已经在一个烂摊子上推进，只能先把代码边界从物理上隔离，避免模块间有任何耦合的可能性，模块之间互相用接口或协议访问。

降低沟通成本

软件工程的特点是自底向上完成总目标的，每个角色都掌握特定的上下文信息，这些信息必须文档化、标准化，才有利于信息低成本的传播。

有了文档化的东西，就要再想办法把信息尽可能的传递给更多的人，就要注重团队内的分析与沟通，平时就要鼓励每个业务模块负责人主动分享自己的模块设计，通过分享倒逼模块负责人不能糊弄、要考虑全面，在互相交流中获得有用的反馈和灵感，同时把好的理念同步传播给尽可能多的人，每个人也都在团队里塑造了个人品牌，增强了彼此的了解和互信，知道彼此的擅长点，之后沟通问题就可以直接找对应的人沟通。

函数式编程

函数式编程的特性与单元测试的目标是一致的，应该采用函数式的思想来编写业务代码（即使因为某些原因无法使用函数式编程语言）。

函数式编程的核心特点是组合，通过一系列函数的组合运算得到目标结果。

能够进行函数组合的前提是，函数是数学意义上的函数，可以从确定的输入得到确定的输出，也就是无副作用，没有状态，这就鼓励多写纯函数，而纯函数是最容易测试的，因为有稳定的预期。

组合的诉求必然导致代码的模块化，代码复用性强才能组合，没有必要存在冗余。

模块化的粒度越小，组合的可能性越多，可以表达的内容更丰富，这样代码就越精简。

业务代码量也少了，测试代码也就少了，测试代码编写速度会显著提升。

业务代码量也少了，要理解的代码量也就少了，并且到处都是原子化的纯函数，接近于数学函数的表达，非常语义化，业务代码可读性也变强了，业务编码速度也会显著提升。

如果能用函数式语言写业务最好，语法上给予直接方便的支持，生产力自然大增。

降低代码理解成本

文档相对还是比较粗化的，实际开发中对代码细节的理解往往也是非常耗时的，经常不知道某段代码是干什么的，为什么要这么写，有什么注意事项，追问也问不到人。

所以代码上要力求让所有人能以最低成本理解内在逻辑，比如：

  • 注释要简洁、完整，用法复杂就最好在注释中配套简单示例。
  • 采用支持函数式特性的现代语言，如Kotlin、Swift，表达力更强、代码量更少，语法更简洁、易用，编码速度和阅读代码速度都会有显著提升。
  • git原子化提交，对每一个代码改动都有简单清晰的理解，减少了梳理庞大代码的工作量。
  • commit message力求结构化的解释清楚每一个改动的原因和影响，git log本身就是一个文档知识库。

减少测试代码量

对于Android、iOS多端开发，功能需求几乎相同的情况下，可以：

  • 业务代码跨平台共用，测试代码只需要写一份，考虑使用Flutter、React Native，或把代码沉降到C层。
  • 测试代码跨平台共用，这种情况一般是黑盒UI测试，考虑使用Appium之类的框架。

测试代码简单化

如果测试代码是能DSL化的，接近自然语言，对于编写者和阅读者都是友好快速的。

普及测试理念

最贴近代码层面的知识传播就是Code Review和结对编程了，通过这两种形式纠正组员的不可测的代码写法。

单元测试的成本会随着测试代码的增多而降低吗？

会降低。

随着测试代码积累的越来越多，业务代码的复用性越来越好，信息流通效率也加强了，大家的能力和意识都越来越强了，针对每个特定业务场景都有了最佳代码测试实践，再写测试代码就是按照模板写了。

单元测试可以测试出所有的问题吗？

不能。

所有的测试用例都是罗列已知的问题，对于未知的问题只有事后发现了再补充到测试用例集里。

对于具有副作用的函数，如文件IO、数据库访问、网络请求，只要副作用能按照预期发生，确定好输入和输出，就可以测到。

能不能写先好代码再补充测试？

理论上可行，前提是你必须已经知道怎么写测试了，时刻保持代码能可测就行，但这没有客观约束，写到不熟悉的业务的时候容易因为赶工而跑偏。

如何在团队中推广单元测试？

因为单测的成本降低是依赖于每个人自底向上的贡献自己在特定业务场景下的解决方案的。

所以要鼓动每个人积极主动、自由的创造发挥，降低信息不对称，加强信息流通效率。

公司需要提供高自由度的环境文化。

符合敏捷开发文化的实践都应该推广。

这又跟公司业务有密切的联系，如果公司的业务发展诉求用敏捷开发比较高效，推广的意愿才强烈，否则就显得没有必要。

依赖注入框架选型

动态注入：

  • JDK内置SPI机制，配合AutoService库可以简化配置。

静态注入：

  • 自己手写，通过构造函数、set函数等方式注入。
  • 有作用域、状态共享等复杂需求采用依赖注入框架：
    • 在Java使用Dagger2。无运行时反射、遵循jsr-330标准。
    • 在Kotlin使用Koin。无辅助代码生成、概念少、API易用。

单元测试怎么写？

三步走

  • 模拟输入
  • 调用执行
  • 验证输出、

确定输入和输出

对于有参数和有返回值的函数，比较好写。

对于没有参数或没有返回值的函数，我们需要人工界定输入和输出是什么，比如某个函数内部调用了某几个特定的方法传递了特定的参数作为这个函数的输出。这就需要mock框架来支持这种验证。

mock、verify、stub、argument capture、spy等概念应运而生，看看Mockito、mockk之类的mock框架的文档就能明白。

androidTest与test

Android环境下测试分为运行在android真机的androidTest，运行在本机JVM的test。

androidTest需要真机，并不方便，编译也占用时间。

所以能在JVM运行所有test最好，所以之前说到要壳化Android代码，业务代码尽可能的不要接触到Android的类。如果业务代码有Android的类和方法会发生什么？因为在JVM上没有Android环境，JVM运行Android API都是直接抛异常的。

Robolectric这个测试框架在JVM环境给出了一个Shadow Android环境，相当于把Android的API在内存中实现了一遍，这样就可以在JVM运行Android代码了。但是运行速度比普通test要慢，因为每次要准备Shadow Android的环境，但速度依然是比androidTest快的。

什么才算一个单元呢？

单元测试的目的是要保证每一个公开的方法都要被测试到。

除非私有方法是多个组合

私有方法作为内部实现可以不写测试，只关注公开的方法的输入和输出。

很多个case都需要的初始化代码放在自定义的Junit Rule中。

一个用例有很多参数用参数化测试。

多线程逻辑怎么写单元测试？

多线程会读取共享状态，并且每一次执行的过程不可重现，所以只能把所有情况做分类讨论，利用锁、CountDownLatch、CyclicBarrier等线程同步工具，让多线程执行逻辑符合特定的预期。

尽量对于线程调用做接口抽象化调度，保证单元测试时可以直接mock线程的调度。

网络请求如何测试？

通过代码拦截或Mock Server模拟数据返回。

数据库如何测试？

模拟数据库环境，再执行数据库操作。

比如Android的JetPack Room提供memory db，Robolectric提供虚拟的SQLite API Shadow环境来模拟所有数据库操作。
