LeetCode.70.爬楼梯（简单）

发表于 2019-10-31 更新于 2021-09-10 分类于 LeetCode Disqus：

题目

LeetCode.70.爬楼梯（简单）

假设你正在爬楼梯。需要 n 阶你才能到达楼顶。
每次你可以爬 1 或 2 个台阶。你有多少种不同的方法可以爬到楼顶呢？
注意：给定 n 是一个正整数。

题解

思路

最优化问题，考虑用动态规划，看是否符合动态规划的条件。
一般从最终状态做倒推，拆解当前状态问题为上一个状态+有限的步骤。

倒推

爬到第n阶，最后一步一定是只有两种可能：

  1. 先爬到第n-1个台阶，再爬1个台阶
  2. 先爬到第n-2个台阶，再爬2个台阶

把两种可能的方法数累加就是爬到第n阶第方法数。

边界处理

爬到第1阶，只有1种爬法，即爬1个台阶。
爬到第2阶，可以从第0阶爬2个台阶，也可以从第1阶爬1个台阶，共2种爬法。

Java8 Lambda表达式

发表于 2019-10-25 更新于 2021-09-10 分类于 Java Disqus：

Java8 Lambda表达式有什么限制？

Java8的lambda和匿名类可以实现类似闭包的功能：它们可以作为参数传递给方法，并且可以访问其作用域之外的变量。

但有一个限制：它们不能修改定义Lambda方法的局部变量的内容。这些变量必须是隐式final的。

之所以会有这些限制，主要是由于局部变量是保存在栈上的，如果Lambda可以直接访问局部变量，且Lambda是在另一个线程使用的，那么使用Lambda的线程可能会在分配该变量的线程将这个变量收回之后，去访问该变量。因此，Java在访问自由局部变量时，实际上访问的是它的副本，而如果局部变量仅会赋值一次那就没什么区别了。

如果允许捕获可变的局部变量，就会引发造成线程不安全的可能性，而这是我们极力希望避免的。基于种种考虑，Java直接在语法层面做出限制，这其实也是不鼓励程序员使用改变外部变量的典型的命令式编程模式。

Java8 Lambda表达式是怎么实现的？

对lambda表达式里的代码，生成静态私有方法，创建一个函数式接口类的实例来调用这个私有静态方法。

这种方式实现的好处在于：

  1. 使用预定义的函数式接口类，不用新增类，避免新类的加载、验证、解析、初始化的耗时。
  2. lambda表达式代码翻译到字节码的过程，可以动态的改变，方便做统一的优化。

为什么Java 8的Lambda表达式的字节码指令要使用invokedynamic指令？

Java8的设计者之一Brian Goetz对于lambda表达式的设计文档中所提到的方案选择：

  • 一是转换策略要足够灵活以能够应对将来的优化。
  • 二是要维持现有class文件标准的稳定性。

用invokedynamic指令干了什么？

invokedynamic调用JDK里的LambdaMetafactory类的方法来把Lambda表达式翻译为字节码。

不用invokedynamic的坏处？

如果把Lambda表达式的翻译方式写死在类的字节码，以后不方便统一修改翻译策略了。

Java8 Lambda表达式为什么不用匿名内部类实现？

Lambda表达式在语法上看起来就跟匿名内部类一样。

但是对于匿名内部类：

  1. 编译器会为每个匿名类生成一个新的class文件。由于每个class文件在使用前都需要经过一系列的加载、验证、解析、初始化的过程，大量的类文件会直接影响应用的启动性能。
  2. 每个新的匿名类都会为类或者接口产生一个新的子类型。如果你为了实现一个比较器，使用了一百多个不同的lambda表达式，这意味着该比较器会有一百多种不同的子类型。这种情况下，JVM的运行时性能调优会变得更加困难。

为什么Java8的特性需要Android Gradle插件特定版本才能支持？

Android系统上，Java-Bytecode（JVM字节码）是不能直接运行在Android系统上的，需要转换成Android-Bytecode（Dalvik/ART 字节码）。

为什么呢？比如Java 8的Lambda表达式是通过invokedynamic指令实现的，Android的dex编译器不支持invokedynamic指令，导致Android不能直接支持Java 8。

既然不能直接支持，那就只能在Java-Bytecode转换到Android-Bytecode这一过程中想办法，间接支持。这个间接支持的过程我们统称为Desugar（脱糖）过程。

具体是怎么实现的呢？比如Java 8的Lambda表达式，javac编译过后，Lambda表达式里的逻辑会被编译为私有静态方法，然后用invokedynamic去在运行时动态创建一个类实例，每个Lambda表达式都会对应一个函数式类接口的，在类方法里调用静态方法。脱糖的过程，就是把invokedynamic指令去除，直接调用静态方法。

脱糖

脱糖即在编译阶段将在语法层面一些底层字节码不支持的特性转换为基础的字节码结构，(比如 List 上的泛型脱糖后在字节码层面实际为 Object)； Android 工具链对 Java8 语法特性脱糖的过程可谓丰富多彩，当然他们的最终目的是一致的：使新的语法可以在所有的设备上运行。

D8编译器 已经支持脱糖，让 Java 8 提供的特性（如 lambdas）可以转换成 Java 7 特性。

D8、R8编译器有什么特别？

D8 编译器作为默认的 DEX 字节码文件编译器，具有更好的性能；

R8 作为 ProGuard 的替代工具，用于代码的压缩（shrinking）和混淆（obfuscation）；

Java 7-8-9 等等新引入的语言特性并不能直接就能用在 Android 开发中，基本上现在的所有的 Android 开发者还在被困在 Java 6 SE 上。

为了让我们能使用上 Java 8 的特性，Google 使用了 Transformation 来增加了一步编译过程叫 desugaring，其实就是将我们代码里使用的 java 8 新特性翻译为 Dalvik/ART 能够识别的 java 6 字节码。这不可避免会导致一个问题 - 更长的编译时间。

为了解决这个问题，在 Android Studio 3.2 中，Google 使用 D8 替换了旧的 dx 编译器。D8 的主要改进是消除 desuguaring 的过程，让其成为 dex 编译的一部分，从而加快编译速度。

单元测试的价值

发表于 2019-09-15 更新于 2021-09-10 分类于 软件测试 Disqus：

衡量一个东西的价值，要看它解决了哪些问题。

我们可以从基本情况出发，逐步推演，发现问题，解决问题，演化到更大的范围，最后从宏观角度观察，综合分析给出结论。

测试驱动开发的价值

先说结论：

对于项目结果而言，TDD带来的最大价值是减少消除信息不对称，消除不确定性，提前发现问题，保证项目能准确的按照预期进行下去，带来更少的回归错误。

对于开发而言，直接好处是会有更少的调试时间、可以更放心的修改代码；间接上，编码效率指数级提升，因为代码架构清晰、可读性强、需求细节明确、一切文档化，信息传递成本很低，只需要理解最少的信息就可以迅速完成任务。

对于测试而言，从低效率手工重复测试工作中解放出来，把精力放在更专业化的兼容性测试、压力测试、自动化模拟场景测试等工作上。

对于产品而言，可以很快看到结果，快速验证想法，迅速得到反馈做出调整。

测试过程的沉淀

对于个人而言，单元测试相当于通过编写代码对功能逻辑进行自测，可以做到：

  • 有及时的测试反馈，通过mock数据环境，快速验证逻辑正确性。
  • 明确所有已知的边界情况，提前规避错误的发生。
  • 沉淀测试过程，测试代码即文档。
  • 测试代码可以自动化执行，别人任何人想验证功能正确性是不需要再花时间了解功能细节的，节约时间。

好处这么多，了所有人的测试时间。

代码测试效率秒杀手工测试。

如果接下来如果尝试给现有的业务代码编写单元测试，会发现：第一个问题：

  • 测试代码会有大量的冗余和重复，不得不复制粘贴很多重复的测试代码，非常痛苦。

为什么会产生大量重复的测试代码？

因为原来的业务代码，本身就是重复冗余的，没有抽象复用，设计不合理，甚至往往产品需求都没有梳理完整，边界情况都没有考虑全。。自然配套的自然测试代码也跟着不合理。

怎样可以不产生重复的测试代码？

那就应该把原来的业务代码，重写为可复用的、标准化、模块化的组织形式。

写到最后，会发现业务代码就完全符合开闭原则了，没有刻意去用什么设计模式，但是代码自然就遵循了所有的设计模式。

那么与其每次重写代码，为什么不在一开始就设计好代码结构？

源头上的诉求

为什么一开始的业务代码没有做到标准化、模块化？

代码是产品设计的实现，如果是因为产品需求、UI设计就没有标准化、模块化、可复用，那么代码实现自然就会理解偏差，每个人都有自己的一套标准。

特别是同一套UI和功能的多端开发，一定要想办法减少信息理解偏差，否则最后极有可能多端实现不一致，等到测试或上线后才发现，费时费力。

现实开发中，很多时候都是开发人员和测试人员去帮，代码只是产品和设计梳理逻辑细节，这是非常本末倒置的的实现。

所以产品设计时就一定要标准化、模块化，并且列明每个业务的各种细节用例，明确每个用例。
并且对于业务功能，需要列出每个业务的输入和、输出，边界情况如何处理。
如果由开发人员来梳理这些事情，那是超出开发的职责范围的，长期下来也是做不到的。

最理想的情况是，只要产品和设计产出标准化的按照某个标准产出某功能的所有测试用例文档，再通过某种工具，直接把文档化的测试用例转化为测试代码的空框架，每一个测试用例对应一个测试方法，方法生成时是空实现，实际由开发填充自行实现。

这样信息不同步的情况是可以减少的。

把源头的问题解决好后，接下来的问题就是：

业务代码会如何演变？

可以发现，为了不让测试代码重复，最终会倒逼业务代码必须要完全符合开闭原则。

并没有刻意去用什么设计模式，代码自然会遵循了所有的设计模式。

分层架构

代码必须要分层架构，否则代码就不能复用，一点小改动会影响很多模块。

不分层的一个典型问题是，多数人喜欢把UI要用的实体类、数据库用的实体类、网络用的实体类三者合一，图方便使用同一个类，因为三种场景下属性含义有重复，看似节约了短期的工作量，实际是在埋坑。

  • 因为一旦你要把网络、数据库、UI的代码独立抽离出来，这一个实体类还要再复制两份。
  • 并且对于单独的模块，这个总的实体类中大概率都会有用不到的属性代码，留着只会增加理解成本、增加程序体积。
  • 实体类的往往还有一堆注解，是网络请求所使用的序列化框架的注解和数据库ORM框架的注解，所以你复制完后，还要去掉注解，否则就得让数据库模块去依赖序列化框架，或者网络模块去依赖ORM框架。

这还是只是一个实体类带来的重构问题，如果有几百个类都是这样，重构阻碍就放大了不知道多少倍，人力损耗是巨大的。那么就应该在一开始就单独定义各自模块的实体类。

带UI的页面要用MVVM架构，因为VM和V自动映射，映射的过程是框架写好的，是标准化的，不需要自己手写；如果是MVP结构，就要手写UI和数据的映射代码，只要手写了，你就得写测试代码覆盖这些逻辑，这是可消除的重复代码，所以MVVM是比MVP少很多测试代码的。

单一职责与依赖注入

类之间的职责必须单一。

比如类A如果要使用到类B和类C中的方法，你是不能在类A里直接创建类B和类C的，否则你无法对类A的方法进行模拟测试，因为无法创建B和C的模拟类传给A使用。

解决这个问题就必须要做依赖注入，把类B和类C的创建移出类A之外，通过set、构造函数等方式传递进来，保证在测试时可以随时替换类B和类C。

可以发现只是为了方便测试做了类隔离，也方便了在开发中替换B和C的实现，代码重构代价更小了。

当依赖注入的需求变得复杂过后，比如要做生命周期管理、状态共享，就需要借助依赖注入框架来完成。

壳化平台特定的代码

在Android中，为了能在JVM上快速运行测试，还会把所有Android相关的类访问都做一层代理封装，并且封装的API是语义化的，这样业务代码既有可读性，又强迫了代码隔离。

Android的类主要充当容器和交互的作用，所有的业务逻辑都可以在JVM上直接模拟测试，不依赖于Android环境的调试测试是非常方便的。

Android代码壳化后，理论上非Android相关的代码应该是可以跨端使用的，只要有工具能做到，就可以节约跨端开发成本。

我们可以把没有重复测试代码的业务代码，称为高可测性的代码。

如何加速测试代码的编写？

反过来思考，阻碍测试代码快速编写的原因有哪些？把这些影响因素都解决了，测试编写就更快了。

有实际经验就知道，一个非常大的阻碍在于沟通成本，有问题是不知道问谁，知道是谁它可能各种原因不能及时给你反馈，信息也不充足，肯定是不如
一定要让沟通成本降到最低，否则会很痛苦。

从另一个角度看，客户端跨端框架可以减少多余端的测试代码的编写，也是有利于提速业务开发进度的。

测试框架本身多端使用。

原子化提交，保证代码可以随时撤回，最小化定位问题看似对产品和设计造成了工作量的增加，其实是一劳永逸的，因为所有的细节都标准化、文档化、自动化了，信息损耗是很小的，任何人不清楚，就查文档，文档没有就补，补完了，上游每一次变动下游都能迅速反应出来。

高可测性的代码还有什么额外的好处？

  • 代码边界清晰了，模块之间是非常独立的，可以建立有效的模块责任制。
  • 问题定位更精确了，很多事情就可以自动化操作了。比如通过日志直接追责责任人。

有了代码质量的客观评判基准，任何新技术框架的使用，不管这么弄，只要能可测，就没有大问题。

客观的流程约束，加强了效率普及。

让人能够更加专注的做事。

单元测试通过、覆盖率100%的代码一定没有问题吗？

比如，有些测试用例一开始没有考虑到怎么办？
单元测试只能对已知的测试用例做处理。
等到程序出错时，发现有某个逻辑没有考虑到，就再补上这条测试用例，将其自动化。

所以还要配合其他测试手段，来全面验证程序的是否存在未知的问题，如压力测试、随机测试、冒烟测试、alpha测试、beta测试等。

对于已知问题，我们应该相信是没有问题的，由责任人保证结果。
