单元测试的价值

发表于 2019-09-15 更新于 2021-09-27 分类于 软件测试 Disqus：

衡量一个东西的价值，要看它解决了什么问题。

测试驱动开发的价值

对于项目结果而言，TDD消除了不确定性，保证项目能比较准确的按照预期推进。

对于开发而言，会有更少的调试时间、更少的回归错误、可以更放心的修改代码，编码效率指数级提升，因为只需要理解最少的信息就可以迅速完成任务。

对于测试而言，从低效率手工重复测试工作中解放出来，把精力放在更专业化的兼容性测试、压力测试、性能测试等工作上。

对于产品而言，可以很快看到结果，快速验证想法，迅速得到反馈做出调整。

测试过程的沉淀

对于个人而言，单元测试相当于通过编写代码对功能逻辑进行自测，可以做到：

  • 及时的测试反馈，通过mock数据环境，快速验证逻辑正确性。
  • 测试代码即文档。
  • 可自动化执行。

接下来如果尝试给现有的业务代码编写单元测试，会发现测试代码会有大量的冗余和重复，不得不复制粘贴很多重复的测试代码。

为什么会产生大量重复的测试代码？

因为原来的业务代码，本身就是重复冗余的，没有抽象复用，设计不合理，甚至往往产品需求都没有梳理完整，边界情况都没有考虑全，测试代码自然就没办法写好。

那就把原来的业务代码，重写为可复用的、标准化、模块化的组织形式。

与其每次重写代码，为什么不在一开始就设计好代码结构？

源头上的诉求

为什么一开始的业务代码没有做到标准化、模块化？

代码是产品设计的实现，如果是因为产品需求、UI设计就没有标准化、模块化、可复用，那么代码就无法标准化。每个功能的开发人员就会按照各自理解封装自己的一套逻辑，都是重复劳动。再加上Android、iOS这种多端开发都是同一套UI和需求，重复劳动的损耗又要多一倍。

要想效率最高，产品设计一开始就要标准化、模块化，明确业务的每个细节，用便于下游理解和处理的形式文档化，比如用测试用例的形式描述需求和设计细节。 在理想情况下，可以通过工具把需求文档生成测试代码模板，多端信息就同步了，CI也可以做强制约束了。

把源头的问题解决好后，接下来可以观察：

业务代码将会如何演变？

为了不让测试代码重复，最终会倒逼业务代码要符合开闭原则，自然遵循所有的设计模式。

分层架构

代码不分层就不能复用，一点小改动就会影响到很多其他模块。

一个常见的例子是，很多人习惯把UI要用的实体类、数据库用的实体类、网络用的实体类使用同一个类，因为三种场景下属性含义有重复，定义三个类麻烦了，但这样定义对于模块化是不方便的：

  • 一旦要把网络、数据库、UI的代码独立出来，这一个实体类还要再复制两份。
  • 对于单独的模块，这个总的实体类中大概率都会有用不到的属性代码，不删除就只会增加理解成本、增加程序体积。
  • 实体类里往往还有一堆注解，一般是序列化框架和ORM框架的注解，所以你复制完实体类后，还要去掉注解，否则就得引入对应框架的依赖才能解决编译报错。

如果有几百个实体类都是这样，重构代价就放大了，所以一开始写业务就要考虑分层模块化。

带UI的页面要用MVVM架构，因为VM和V自动映射，映射的过程是框架写好的、是标准化的，不需要自己手写；如果是MVP结构，就要手写UI和数据的映射代码，只要手写了，你就得写测试代码覆盖这些逻辑，所以MVVM是比MVP少一些模板化的测试代码的。

MVVM只是一个分层的基本思想，也不是说就只分三层，如果只是单纯的把Activity、Fragment的逻辑处理代码换个位置放到ViewModel层，测试代码还是很难写。

实际可以有无数层嵌套，每一个功能职责都是一层，每一层之间有交互，跨层之间无感知。每一层不管是叫UserCase、Domain、Model、Repository还是什么，只要复用性强，不会产生重复测试代码，就是好的分层。

单一职责与依赖注入

类之间的职责必须单一。

比如有2个类A和B，A的方法如果要使用到B方法，如果在A里直接创建B的实例是不好做测试的，因为要替换B的实例就要修改A的代码，怎么样能不修改代码又能模拟B的行为呢？那就是不要在A里创建B，把创建B的过程放在其他地方，把B的实例直接传给A使用，这样模拟B的行为就不用修改A的代码了。

这就是依赖注入，通过set函数或构造函数等方式传递依赖的对象。

如果有生命周期管理、状态共享的复杂注入需求，注入的逻辑就有重复了，就需要借助依赖注入框架来完成。

可以看到，为了方便测试，强迫了类的隔离和复用。

代码复用性越高，所有人每次写功能就要找有没有可以复用的代码，所以就要给代码加很多注释和文档，API也要语义化、简单化，不然信息检索成本和沟通成本就高了。

壳化平台代码

在Android中，为了能在JVM上快速运行测试，还会把所有Android相关的类访问都做一层代理封装，这样又强化了代码隔离，Android代码就壳化了。

这有点跨端框架的味道。非平台相关的代码如果能够统一，就能节约开发成本。

如何加速测试代码的编写？

反过来思考，阻碍测试代码编写的原因还有哪些？把这些影响因素都解决了就好了。

代码隔离

如果已经在一个烂摊子上推进模块化，只能先把代码边界从物理上隔离，避免后续再有任何代码耦合的可能性，模块之间互相用接口或协议互通访问，在新业务上把新的流程走起来，老业务等到需求迭代时再重写。

函数式编程

函数式编程的核心特点是组合，通过一系列函数的组合运算得到目标结果。

函数组合的前提是，函数得是纯的，即可以从确定的输入得到确定的输出，也就是无副作用、没有状态，纯函数是最容易测试的。

组合的诉求必然导致代码的模块化，代码复用性强才能组合，没有必要存在冗余。

模块化的粒度越小，组合的可能性越多，可以表达的内容更丰富，这样代码就越精简。

业务代码量也少了，测试代码也就少了，测试代码编写速度会提升。

业务代码量也少了，要理解的代码量也就少了，并且到处都是原子化的纯函数，接近于数学函数的表达，非常语义化，业务代码可读性也变强了，业务编码速度也会提升。

函数式编程的诉求与单元测试的目标是一致的，所以尽可能用函数式的思想来编写业务代码。

降低信息流通成本

代码复用性增强后，要保证每块代码的理解成本尽可能的低，才能加速复用的过程。比如：

  • 注释要简洁、完整，复杂的可以配套简单示例。
  • git原子化提交，对每一个代码改动都有清晰直白的描述，积累代码知识库。
  • Code Review严格规范，传递最佳实践。

减少测试代码量

对于Android、iOS多端开发，功能需求几乎相同的情况下，可以：

  • 业务代码跨平台共用，测试代码只需要写一份，考虑使用Flutter、React Native，或把代码沉降到C层。
  • 测试代码跨平台共用，这种情况一般是黑盒UI测试，考虑使用Appium之类的框架。

测试代码简单化

如果测试代码是能DSL化的，接近自然语言，对于编写和阅读都是友好快速的。

单元测试的成本会随着测试代码的增多而降低吗？

会降低。

随着测试代码积累的越来越多，业务代码的复用性越来越好，信息流通效率也加强了，大家的写测试也越来越熟练了，针对每个特定业务场景都有了最佳代码测试实践，再写测试代码就是按照模板写了。

单元测试可以测试出所有的问题吗？

不能。

所有的测试用例都是罗列已知的问题，对于未知的问题只有事后发现了再补充到测试用例集里。

如果本身对于需求理解就是错的，写的测试也是错的。所以还需其他测试方法来发现未考虑到的问题。

能不能写先好代码再补充测试？

理论上可行，前提是你必须已经知道怎么写测试了，时刻保持代码能可测就行，但这没有客观约束，写到不熟悉的业务的时候容易因为赶工而跑偏。

如何在团队中推广单元测试？

两个方向：降低阻碍、放大价值。

降低阻碍

软件开发中，每个人都掌握着特定的上下文信息，项目整体是依赖于每个人的贡献，所以新的方法论和思想理念需要低成本、完整的传达到每个人，让每个人都能迅速掌握。

这需要无障碍的交流环境，把问题暴露出来，别人才能快速响应解决，加强互助协作的氛围。

放大价值

先让几个强人在小范围以最低成本把流程跑起来，解决各种公共基本问题，带来显著收益后，在逐步推广到其他更复杂的业务中，拉入新的人参与，通过规范约束、Code Review等形式普及最佳实践，让每个人感受到确实可以用很低的学习成本带来很大的收益，学习新东西的意愿就强了，有价值的东西会逐渐形成口碑传播出去。

依赖注入框架选型

动态注入：

  • JDK内置SPI机制，配合AutoService库可以简化配置。

静态注入：

  • 自己手写，通过构造函数、set函数等方式注入。
  • 有作用域、状态共享等复杂需求采用依赖注入框架：
    • 在Java可以使用Dagger2。无运行时反射、遵循jsr-330标准。
    • 在Kotlin可以使用Koin。无辅助代码生成、概念少、API易用。

单元测试怎么写？

三步走

  • 模拟输入
  • 调用执行
  • 验证输出

确定输入和输出

对于有参数和有返回值的函数，比较好写，确定好输入和输出，直接断言验证。

对于没有参数或没有返回值的函数，需要人工界定输入和输出是什么，比如某个函数内部调用了某几个特定的方法传递了特定的参数作为这个函数的输出，Junit就不够用了，需要通过代理要模拟的对象来做验证，但自己写代理逻辑会有不少重复劳动，这就需要标准化的mock框架来做验证。

mock、stub、spy、verify、参数捕获等概念就应运而生。

mock框架应该选哪个？

选取原则：

  • 学习成本要低，这样才利于普及，不给开发增加额外负担。
  • API可读性要高，越DSL化越好，接近自然语言越好。
  • 功能要全面，最好一个库能满足绝大多数测试的需求，如果要引用多个库更新起来是比较麻烦的，配套使用可能也会存在不兼容。

Java下的mock框架有EasyMock、JMockit、Mockito、PowerMock等。

Mockito和EasyMock对比：Mockito vs EasyMock

Mockito和JMockit对比：Difference between mockito and jmockit

整体来说，Mockito是最好用的，API简单、易读，JMockit和EasyMock的学习成本比较高。

Mockito的实现方式是新建一个类继承要mock的类再改写要mock的方法，但是有的方法无法继承，比如final、private、static方法。

PowerMock通过修改字节码解决Mockito不能mock static、final、private方法的问题，但是API可读性比较差。

Kotlin下使用mockk做mock会更方便，符合Kotlin语法习惯，可读性也更好。

androidTest与test

Android环境下测试分为运行在android真机的androidTest、运行在本机JVM的test。

androidTest运行需要真机，不太方便，编译也相对耗时。

JVM的test的特点是运行速度快，缺点是没有Android环境，运行Android的方法会直接抛异常。

能在JVM运行所有测试用例效率和方便程度是最高的。所以之前就说到要壳化Android代码，业务代码尽可能的不要接触到Android的类。

Robolectric在JVM下模拟了Android环境，相当于把Android的API在内存中实现了一遍，这样就可以在JVM运行Android代码了。不过运行速度比普通test要慢，因为每次要准备Shadow Android的环境，但速度依然是比androidTest快的。有一些Android代码需要测试可以考虑使用Robolectric。

单元测试粒度需要多细？

没有硬性的标准，原则就是保证关键代码逻辑分支都走到了，测试代码尽可能少的重复冗余。

什么是关键代码逻辑？

最少要对主要的public方法做测试，因为public方法会被其他模块使用到，需要保证逻辑的正确性。

也就是说私有方法不一定要单独写测试，但是不代表私有方法的逻辑不会走到，只是可以不用要求让每个私有方法都单独对应一个测试方法。

JavaBean中的set、get这些模板的public方法，其实没有必要特意写测试，如果不放心就让模板化的东西让测试也能模板化的生成就行了。

所有逻辑复杂、可能会产生严重错误的地方都是要测试到的。

每个测试用例都有重复的初始化代码怎么抽象？

Junit支持@Before、@After、@BeforeClass、@AfterClass等注解，用于case执行前后公共逻辑的执行。

跨测试类的公共逻辑可以封装为自定义的Junit Rule。

输入参数有很多种组合怎么办？

用参数化测试。

多线程逻辑怎么写单元测试？

每个线程的逻辑需要先单独测试。

多线程只测试线程调度逻辑是否正确，如果涉及到内部细节的测试，测试代码的复杂度会爆炸。

比如通过线程池调度某些方法，那么线程池最好是通过注入接口的形式给调用处使用，这样就可以在测试时模拟线程池调度，测试会很方便。

由于多线程会读取共享状态，并且每一次执行的过程不可重现，所以只能把所有可能出现的情况做分类讨论。

再利用锁、CountDownLatch、CyclicBarrier等线程同步工具，让多线程执行逻辑符合特定的预期。

网络请求如何测试？

保证让网络请求逻辑的API真实的运行过，可以通过代码拦截或Mock Server模拟数据返回，并且这种模拟一定要固定下来，可重复操作，不要每次都搭环境。

数据库如何测试？

模拟数据库环境，再执行数据库操作。

比如Android的JetPack Room提供memory db，Robolectric提供虚拟的SQLite API Shadow环境来模拟所有数据库操作。

断言库如何选择？

对测试代码最大的要求就是API可读性强，学习成本低，这样才能加强工具普及和信息流通。

Java的断言库有Hamcrest、Truth等，符合逻辑推理习惯、语义化的API。

Kolint的断言库有Kluent，借助Kotlin对DSL的支持把API写的跟自然语言一样。

Android的UI怎么测试？

白盒测试框架有Espresso，只能针对有源码的App进行测试，并且只能写在androidTest，要真机运行。

也可以用Robolectric在JVM上测UI。

具体选择哪个看应用场景。

黑盒测试框架有UIAutomator，可以测任意多个App，但是只能Android下使用。

黑盒测试最好是Android和iOS统一使用同一套框架，降低学习成本，Appium可以用在多平台的黑盒UI测试。

黑盒测试可让非开发人员去写测试。

单元测试覆盖率达到多少才算合适？

即使覆盖率100%，也不能保证没有错误，因为还是有可能遗漏掉正确的逻辑。

覆盖率100%也不能代表代码就设计的很好，可以存在大量不合理的代码也能让覆盖率100%。

可读性、模块化都无法从覆盖率角度衡量，其他手段也无法准确衡量。

只能尽可能的遵循一些原则来保证测试性，也就是上面讨论的所有影响代码不可测的情况都要消除掉，即使没有一行测试代码，代码架构也可以是非常好的，这需要依赖人来判断，所以要在团队想办法形成共识。

最好还是写出测试，把脑子里的知识沉淀下来，传播分享出去放大它的价值，具体写多少，看实际收益比。
