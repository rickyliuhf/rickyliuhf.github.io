单元测试的价值

发表于 2019-09-15 更新于 2021-09-14 分类于 软件测试 Disqus：

衡量一个东西的价值，要看它解决了什么问题。

测试驱动开发的价值

结论：

对于项目结果而言，TDD带来的最大价值是消除了不确定性，保证项目能准确的按照预期进行下去。

对于开发而言，会有更少的调试时间、更少的回归错误、可以更放心的修改代码，编码效率指数级提升，因为只需要理解最少的信息就可以迅速完成任务。

对于测试而言，从低效率手工重复测试工作中解放出来，把精力放在更专业化的兼容性测试、压力测试、性能测试等工作上。

对于产品而言，可以很快看到结果，快速验证想法，迅速得到反馈做出调整。

测试过程的沉淀

对于个人而言，单元测试相当于通过编写代码对功能逻辑进行自测，可以做到：

  • 有及时的测试反馈，通过mock数据环境，快速验证逻辑正确性。
  • 明确所有已知的边界情况，提前规避错误的发生。
  • 沉淀测试过程，测试代码即文档。
  • 测试代码可以自动化执行，别人任何人想验证功能正确性是不需要再花时间了解功能细节的，节约时间。

如果接下来如果尝试给现有的业务代码编写单元测试，会发现测试代码会有大量的冗余和重复，不得不复制粘贴很多重复的测试代码，非常痛苦。

为什么会产生大量重复的测试代码？

因为原来的业务代码，本身就是重复冗余的，没有抽象复用，设计不合理，甚至往往产品需求都没有梳理完整，边界情况都没有考虑全，自然配套的自然测试代码也跟着不合理。

怎样可以不产生重复的测试代码？

那就应该把原来的业务代码，重写为可复用的、标准化、模块化的组织形式。

写到最后，会发现业务代码就完全符合开闭原则了，没有刻意去用什么设计模式，但是代码自然就遵循了所有的设计模式。

那么与其每次重写代码，为什么不在一开始就设计好代码结构？

源头上的诉求

为什么一开始的业务代码没有做到标准化、模块化？

代码是产品设计的实现，如果是因为产品需求、UI设计就没有标准化、模块化、可复用，那么代码就无法标准化。

每个功能的开发人员必然会按照各自理解封装自己的一套逻辑，都是重复劳动。

再加上Android、iOS这种多端开发都是同一套UI，重复劳动的损耗又要多一倍。

要想效率最高，产品设计一开始就一定要标准化、模块化，明确业务的每个细节，用便于下游理解和处理的形式文档化，比如用测试用例的形式描述需求。 这样在理想情况下，可以通过工具把产品设计侧提供的标准化的测试用例直接转换为开发侧的测试代码模板，减小了开发编写测试用例的负担。

开发如果没有去填充测试代码，CI跑测试就会报错，不能允许出包，这就对开发做了强制约束，降低了需求遗漏和理解不一致的可能性。

在Android、iOS多端开发下，信息都能第一时间及时同步，避免了多端实现不一致。

看似增加了产品和设计的工作量，其实最终是节省了所有人的时间，上下游所有人都将受益。否则巨大的沟通成本和信息不同步会把所有人互相消耗的精疲力尽，结果问题还没有从根本上解决。

把源头的问题解决好后，接下来可以观察：

业务代码将会如何演变？

可以发现，为了不让测试代码重复，最终会倒逼业务代码必须要完全符合开闭原则。

并没有刻意去用什么设计模式，代码自然的就遵循了所有的设计模式。

分层架构

代码必须要分层架构，否则代码就不能复用，一点小改动就会影响到很多其他模块。

一个常见的例子是，很多人喜欢把UI要用的实体类、数据库用的实体类、网络用的实体类三者合一使用同一个类，因为三种场景下属性含义有重复，定义三个类看起来麻烦了，但对于模块化是不方便的：

  • 一旦你要把网络、数据库、UI的代码独立抽离出来，这一个实体类还要再复制两份。
  • 对于单独的模块，这个总的实体类中大概率都会有用不到的属性代码，不删除就只会增加理解成本、增加程序体积。
  • 实体类的往往还有一堆注解，是网络请求所使用的序列化框架的注解和数据库ORM框架的注解，所以你复制完后，还要去掉注解，否则就得让数据库模块去依赖序列化框架，或者网络模块去依赖ORM框架。

这只是一个实体类带来的重构问题，如果有几百个类都是这样，重构阻碍就放大了很多倍，所以在一开始就要按分层模块化的标准来编写代码。

带UI的页面要用MVVM架构，因为VM和V自动映射，映射的过程是框架写好的、是标准化的，不需要自己手写；如果是MVP结构，就要手写UI和数据的映射代码，只要手写了，你就得写测试代码覆盖这些逻辑，所以MVVM是比MVP少一些模板化的测试代码的。

单一职责与依赖注入

类之间的职责必须单一。

比如类A如果要使用到类B和类C中的方法，你是不能在类A里直接创建类B和类C的，否则你无法对类A的方法进行模拟测试，因为无法创建B和C的模拟类传给A使用。

解决这个问题就必须要做依赖注入，把类B和类C的创建移出类A之外，通过set、构造函数等方式传递进来，保证在测试时可以随时替换类B和类C。

可以发现只是为了方便测试做了类隔离，也方便了在开发中替换B和C的实现，代码重构代价更小了。

当依赖注入的需求变得复杂，比如要做生命周期管理、状态共享，就需要借助依赖注入框架来完成。

壳化平台特定的代码

在Android中，为了能在JVM上快速运行测试，还会把所有Android相关的类访问都做一层代理封装，这样强迫了代码隔离。并且封装的API是语义化的，业务代码又增加了可读性。

Android的类主要充当容器和交互的作用，所有的业务逻辑都可以在JVM上直接模拟测试，是非常方便的。

Android代码壳化后，理论上非Android相关的代码应该是可以跨端使用的，只要有工具能做到。

我们可以简单的把没有重复测试代码的业务代码，称为高可测性的代码。

如何加速测试代码的编写？

反过来思考，阻碍测试代码编写的原因还有哪些？把这些影响因素都解决了就好了。

提前做标准化设计

之前说了如果业务代码架构设计不合理，复用性很差，测试代码会很难写，要被迫修改业务代码，重构是非常大的成本。

如果已经在一个烂摊子上推进，只能先把代码边界从物理上隔离，避免模块间有任何耦合的可能性，模块之间互相用接口或协议访问。

降低沟通成本

软件工程的特点是自底向上完成总目标的，每个角色都掌握特定的上下文信息，这些信息必须文档化、标准化，才有利于信息低成本的传播。

有了文档化的东西，就要再想办法把信息尽可能的传递给更多的人，就要注重团队内的分析与沟通，平时就要鼓励每个业务模块负责人主动分享自己的模块设计，通过分享倒逼模块负责人要考虑全面，在互相交流中获得有用的反馈，同时把好的理念同步传播给尽可能多的人，增强了彼此的了解和互信，之后沟通会越来越顺畅。

函数式编程

函数式编程的核心特点是组合，通过一系列函数的组合运算得到目标结果。

能够进行函数组合的前提是，函数是数学意义上的函数，可以从确定的输入得到确定的输出，也就是无副作用，没有状态，这就鼓励多写纯函数，而纯函数是最容易测试的，因为有稳定的预期。

组合的诉求必然导致代码的模块化，代码复用性强才能组合，没有必要存在冗余。

模块化的粒度越小，组合的可能性越多，可以表达的内容更丰富，这样代码就越精简。

业务代码量也少了，测试代码也就少了，测试代码编写速度会显著提升。

业务代码量也少了，要理解的代码量也就少了，并且到处都是原子化的纯函数，接近于数学函数的表达，非常语义化，业务代码可读性也变强了，业务编码速度也会显著提升。

函数式编程的特性与单元测试的目标是一致的，应该采用函数式的思想来编写业务代码（即使因为某些原因无法使用函数式编程语言）。

如果能用函数式语言写业务最好，语法上给予直接方便的支持，生产力自然大增。

降低代码理解成本

文档相对还是比较粗化的，实际开发中对代码细节的理解往往也是非常耗时的，经常不知道某段代码是干什么的，为什么要这么写，有什么注意事项，追问也问不到人。

所以代码上要力求让所有人能以最低成本理解内在逻辑，比如：

  • 注释要简洁、完整，用法复杂就最好在注释中配套简单示例。
  • 采用支持函数式特性的现代语言，如Kotlin、Swift，表达力更强、代码量更少，语法更简洁、易用，编码速度和阅读代码速度都会有显著提升。
  • git原子化提交，对每一个代码改动都有简单清晰的理解，减少了梳理庞大代码的工作量。
  • commit message力求结构化的解释清楚每一个改动的原因和影响，git log本身就是一个文档知识库。

减少测试代码量

对于Android、iOS多端开发，功能需求几乎相同的情况下，可以：

  • 业务代码跨平台共用，测试代码只需要写一份，考虑使用Flutter、React Native，或把代码沉降到C层。
  • 测试代码跨平台共用，这种情况一般是黑盒UI测试，考虑使用Appium之类的框架。

测试代码简单化

如果测试代码是能DSL化的，接近自然语言，对于编写者和阅读者都是友好快速的。

普及测试理念

最贴近代码层面的知识传播就是Code Review和结对编程了，通过这两种形式纠正组员的不可测的代码写法。

单元测试的成本会随着测试代码的增多而降低吗？

会降低。

随着测试代码积累的越来越多，业务代码的复用性越来越好，信息流通效率也加强了，大家的能力和意识都越来越强了，针对每个特定业务场景都有了最佳代码测试实践，再写测试代码就是按照模板写了。

单元测试可以测试出所有的问题吗？

不能。

所有的测试用例都是罗列已知的问题，对于未知的问题只有事后发现了再补充到测试用例集里。

如果本身对于测试用例理解就是错的，你写的测试验证也是错的。所以还需黑盒测试、内测、公测等方式发现未考虑到的问题。

对于具有副作用的函数，如文件IO、数据库访问、网络请求，只要副作用能按照预期发生，确定好输入和输出，就可以测到。

能不能写先好代码再补充测试？

理论上可行，前提是你必须已经知道怎么写测试了，时刻保持代码能可测就行，但这没有客观约束，写到不熟悉的业务的时候容易因为赶工而跑偏。

如何在团队中推广单元测试？

单测的成本降低是依赖于每个人自底向上的贡献自己在特定业务场景下的解决方案的，需要全员参与，讲究团队协作。

需要让每个人自由发挥，实现手段上不要有太多限制，因为特定上下文环境下解决问题的方法各不相同，在思想和理念上形成共识即可。

需要招聘能力强、自我驱动、全局思维的员工，可以加强团队协作效果，至少要有几个核心的推广者。

几个核心的推广者需要现在几个好下手的业务上，把完整的流程跑起来，能看到不错的效果，解决公共基础障碍，这样别人就有了参照。

接下来就是要分享交流，把有价值的思想观念传播出去，每个业务案例也都讲解分享，尽可能的降低每个人上手的难度，认知普及了，主要障碍也没有了，事情就好做了。

所以需要解决信息流通障碍，鼓励随时随地面对面沟通，部门内、部门级、上下级间都要制造交流机会，保证能够第一时间解决任何实际问题，提高整体效率。

最后其实还是要看公司的业务特点，如果业务不需要这些方法论就可以搞的不错，那这些方法论自然就没有迫切的需求被推广了。

公司实际的文化，就是实际业务问题的惯用的解决方案的总集合，而不是嘴上宣传的。

依赖注入框架选型

动态注入：

  • JDK内置SPI机制，配合AutoService库可以简化配置。

静态注入：

  • 自己手写，通过构造函数、set函数等方式注入。
  • 有作用域、状态共享等复杂需求采用依赖注入框架：
    • 在Java使用Dagger2。无运行时反射、遵循jsr-330标准。
    • 在Kotlin使用Koin。无辅助代码生成、概念少、API易用。

单元测试怎么写？

三步走

  • 模拟输入
  • 调用执行
  • 验证输出

确定输入和输出

对于有参数和有返回值的函数，比较好写，Junit可以直接断言验证。

对于没有参数或没有返回值的函数，我们需要人工界定输入和输出是什么，比如某个函数内部调用了某几个特定的方法传递了特定的参数作为这个函数的输出，Junit就不够用了，需要通过代理要mock的对象来做验证，但自己写代理逻辑太冗余了，这就需要标准化的mock框架来做验证。

mock、stub、spy、verify、参数捕获等概念就应运而生。

mock框架应该选哪个？

选取原则：

  • 学习成本要低，这样才利于普及，不给开发增加额外负担。
  • API可读性要高，越接近自然语言，就可以不用看函数的文档直接就能懂在做什么，有利于快速编码，所以API越DSL化越好。
  • 功能要全面，最好一个库能满足绝大多数测试的需求，如果要引用多个库更新起来是比较麻烦的，配套使用可能也会存在不兼容。

Java下的mock框架有EasyMock、JMockit、Mockito、PowerMock等。

Mockito和EasyMock对比：Mockito vs EasyMock

Mockito和JMockit对比：Difference between mockito and jmockit

整体来说，Mockito是最好用的，API简单、易读，JMockit和EasyMock的学习成本比较高。

不过Mockito的实现方式是新建一个类继承要mock的类再改写要mock的方法，但是有的方法无法继承，比如final、private、static方法。

PowerMock通过修改字节码解决Mockito不能mock static、final、private方法的问题，但是API可读性比较差。

Kotlin下使用mockk做mock会更方便一些，mockk专门针对kotlin的场景做了优化处理。

androidTest与test

Android环境下测试分为运行在android真机的androidTest、运行在本机JVM的test。

androidTest运行需要真机，不太方便，编译也相对耗时。

JVM的test的特点是运行速度快，缺点是没有Android环境，运行Android的方法会直接抛异常。

能在JVM运行所有测试用例效率和方便程度是最高的。所以之前就说到要壳化Android代码，业务代码尽可能的不要接触到Android的类。

Robolectric这个测试框架在JVM下模拟了Android环境，相当于把Android的API在内存中实现了一遍，这样就可以在JVM运行Android代码了。不过运行速度比普通test要慢，因为每次要准备Shadow Android的环境，但速度依然是比androidTest快的。有一些Android代码需要测试可以考虑使用Robolectric。

单元测试粒度需要多细？

没有严格的一个标准，只有原则，原则就是保证关键代码逻辑分支都走到了，测试代码没有重复冗余。

什么是关键代码逻辑？最少要对public方法做测试，因为public方法会被其他模块使用到，需要保证逻辑的正确性。

也就是说私有方法不一定要单独写测试，但是不代表私有方法的逻辑不会走到，只是可以不用要求让每个私有方法都单独对应一个测试方法。

JavaBean中的set、get这些模板的public方法，其实没有必要特意写测试，如果不放心就让模板化的东西让测试也能模板化的生成就行了。

所有逻辑复杂、可能会产生严重错误的地方都是要测试到的。

每个测试用例都有重复的初始化代码怎么抽象？

Junit支持@Before、@After、@BeforeClass、@AfterClass等注解，用于case执行前后公共逻辑的执行。

跨测试类的公共逻辑可以封装为自定义的Junit Rule。

输入参数有很多种组合怎么办？

用参数化测试。

多线程逻辑怎么写单元测试？

每个线程的逻辑需要先单独测试。

多线程只测试线程调度逻辑是否正确，如果涉及到内部细节的测试，测试代码的复杂度会爆炸。

比如通过线程池调度某些方法，那么线程池最好是通过注入接口的形式给调用处使用，这样就可以在测试时模拟线程池调度，测试会很方便。

由于多线程会读取共享状态，并且每一次执行的过程不可重现，所以只能把所有可能出现的情况做分类讨论。

再利用锁、CountDownLatch、CyclicBarrier等线程同步工具，让多线程执行逻辑符合特定的预期。

网络请求如何测试？

保证让网络请求逻辑的API真实的运行过，可以通过代码拦截或Mock Server模拟数据返回，并且这种模拟一定要固定下来，可重复操作，不要每次都搭环境。

数据库如何测试？

模拟数据库环境，再执行数据库操作。

比如Android的JetPack Room提供memory db，Robolectric提供虚拟的SQLite API Shadow环境来模拟所有数据库操作。

断言库如何选择？

对测试代码最大的要求就是API可读性强，学习成本低，这样才能加强工具普及和信息流通。

Java的断言库有Hamcrest、Truth等符合逻辑推断、语义化的API。

Kolint的断言库有Kluent，借助Kotlin DSL的语法支持把API自然语言化。

Android的UI怎么测试？

白盒测试框架有Espresso，只能针对有源码的App进行测试，并且只能写在androidTest，要真机运行。

也可以用Robolectric在JVM上测UI。

具体选择哪个看应用场景。

黑盒测试框架有UIAutomator，可以测任意多个App，但是只能Android下使用。

黑盒测试最好是Android和iOS统一使用同一套框架，降低学习成本，Appium可以用在多平台的黑盒UI测试。

黑盒测试可让非开发人员去写测试。

单元测试覆盖率达到多少才算合适？

即使覆盖率100%，也不能保证没有错误，因为还是有可能遗漏掉正确的逻辑。

覆盖率100%也不能代表代码就设计的很好，可以存在大量不合理的代码也能让覆盖率100%。

可读性、模块化都无法从覆盖率角度衡量，其他手段也无法准确衡量。

只能尽可能的遵循一些原则来保证测试性，也就是上面讨论的所有影响代码不可测的情况都要消除掉，即使没有一行测试代码，代码架构也可以是非常好的，这需要依赖人来判断，所以要在团队想办法形成共识。

最好还是写出测试，把脑子里的知识沉淀下来，传播分享出去放大它的价值，具体写多少，看实际收益比。
